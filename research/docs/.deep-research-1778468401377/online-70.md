# Online Research: extensions/types/ — URL and TextEncoder ambient declarations

(no external research applicable)

The scope under examination is `extensions/types/`, which consists of two ambient TypeScript declaration files — `lib.url.d.ts` and `lib.textEncoder.d.ts` — totalling 21 lines of code. These files do nothing more than declare global ambient types for `URL`, `URLSearchParams`, `TextEncoder`, and `TextDecoder`, re-exposing the corresponding interfaces already present in Node's built-in `url` and `util` modules. They contain no runtime logic, no third-party library dependencies, no bundler configuration, and no platform-specific bindings. They exist solely to make TypeScript aware of these globals in execution contexts where the standard lib declarations might be absent or incomplete.

From a porting perspective — moving from a TypeScript/Electron host to a Tauri/Rust host — these declarations carry no migration burden whatsoever. The Web APIs they describe (`URL`, `URLSearchParams`, `TextEncoder`, `TextDecoder`) are part of the WinterCG-aligned runtime surface that both modern browser engines and the WebView2/WebKit webviews embedded by Tauri expose natively. In a Tauri frontend context the TypeScript compiler would pick up these types from its standard `lib.dom.d.ts` or `lib.webworker.d.ts`, making the ambient re-declarations redundant rather than necessary. In any Rust backend code there is no TypeScript compilation at all, so the files are simply irrelevant.
