The scope for this research task is `extensions/cpp/` — a single-file C++ language extension consisting only of TextMate grammar definitions, a `language-configuration.json`, and NVCC problem matchers. This is entirely declarative configuration data with no executable logic, no TypeScript source, no Electron API calls, and no third-party runtime libraries. There is therefore nothing to port in the conventional software sense, and no external research is needed to inform a Tauri/Rust migration.

TextMate grammars are JSON or plist payloads that define syntax-highlighting rules via regular-expression patterns and scope names. They are editor-agnostic by design: the same `.tmLanguage` / `.tmGrammar` file can be consumed by VS Code, Sublime Text, Atom, Neovim (via nvim-treesitter fallback layers), or a Rust-based host using the `syntect` crate, which reads the identical format. A Tauri port that uses `syntect` for syntax highlighting would ingest the existing grammar files unchanged. A port that adopts Tree-sitter instead would need a separate C++ Tree-sitter grammar, but that is a choice about the target host's highlighting architecture, not a consequence of anything in the extension itself.

`language-configuration.json` describes editor behavior such as bracket pairing, comment token characters, auto-closing pairs, and word-pattern rules. It is likewise pure data. Whatever language-service layer is built into a Tauri-based editor would either parse this file directly or provide an equivalent configuration mechanism; either way the file's contents are trivially readable and require no migration analysis.

NVCC problem matchers are JSON schemas that describe how to parse compiler diagnostic output into structured editor annotations. They have no runtime code and no dependencies. Their migration footprint is confined to ensuring the target build-task infrastructure supports the same matcher format or provides an equivalent.
