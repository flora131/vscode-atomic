### Per-File Notes

#### `extensions/json/build/update-grammars.js`

This is a pure Node.js build script invoked via `npm run update-grammar` (declared at `package.json:12`). It has no runtime presence in VS Code itself.

**Key symbol:** `adaptJSON` (lines 9–32)

`adaptJSON(grammar, name, replacementScope, replaceeScope = 'json')` performs an in-memory mutation of a downloaded TextMate grammar object before it is written to disk:

- Line 10: sets `grammar.name` to the supplied display name.
- Line 11: sets `grammar.scopeName` to `"source" + replacementScope` (e.g., `"source.json.comments"`).
- Lines 12–26: builds a `RegExp` that matches `"." + replaceeScope` globally, then recurses through every rule node in the grammar tree. For each rule, the `name` and `contentName` string fields are rewritten by replacing every occurrence of the old scope fragment with `replacementScope`. The recursion is depth-first via `for…in` over object properties (line 20–25).
- Lines 28–31: iterates the top-level `grammar.repository` map and applies `fixScopeNames` to each entry.

**Key symbol:** `updateGrammar.update` calls (lines 35–39)

Four calls are made to the external `vscode-grammar-updater` library:

1. Line 35 — downloads `JSON.tmLanguage` from `microsoft/vscode-JSON.tmLanguage` and writes it verbatim to `./syntaxes/JSON.tmLanguage.json`.
2. Line 36 — downloads the same source file, runs `adaptJSON` with `name="JSON with Comments"`, `replacementScope=".json.comments"` (default `replaceeScope="json"`), writes to `./syntaxes/JSONC.tmLanguage.json`.
3. Line 37 — downloads again, runs `adaptJSON` with `name="JSON Lines"`, `replacementScope=".json.lines"`, writes to `./syntaxes/JSONL.tmLanguage.json`.
4. Line 39 — downloads `autogenerated/jsonc.tmLanguage.json` from `jeff-hykin/better-snippet-syntax`, runs `adaptJSON` with `name="Snippets"`, `replacementScope=".json.comments.snippets"`, `replaceeScope="json.comments"`, writes to `./syntaxes/snippets.tmLanguage.json`.

The script produces all four grammar files from two upstream sources. The three derived grammars (JSONC, JSONL, snippets) share the structural token rules of the base grammar; only scope-name strings differ.

#### `extensions/json/language-configuration.json`

Configures editor mechanics for all four language IDs (each `language-configuration.json` reference in `package.json` points to this same file).

**`comments`** (lines 2–8): declares `//` as line comment and `/* */` as block comment. This is consumed by JSONC-aware tooling; standard JSON technically has no comments, but the file is shared across all four language IDs.

**`brackets`** (lines 9–13): declares `{}`  and `[]` as matching bracket pairs for bracket-matching and fold detection.

**`autoClosingPairs`** (lines 19–63): six pairs — `{}`, `[]`, `()`, `''`, `""`, and backtick pairs. The `notIn` constraints prevent auto-close from firing inside `"string"` tokens; `""` and backtick additionally exclude `"comment"` contexts.

**`indentationRules`** (lines 65–68):
- `increaseIndentPattern` (line 66): a regex that matches a line ending with an open `{` or `[` that is not inside a string (accounts for escaped quotes and balanced string pairs).
- `decreaseIndentPattern` (line 67): `^\s*[}\]],?\s*$` — a line consisting only of a closing brace or bracket with optional comma and surrounding whitespace.

**`onEnterRules`** (lines 69–83): one rule. When the cursor is on a line matching `//.*` (`beforeText`) and the `afterText` is non-empty, pressing Enter appends `"// "` to the new line with `indent: "none"`. This continues single-line comment blocks across newlines.

### Cross-Cutting Synthesis

**Extension purpose.** `extensions/json` is a pure declarative extension. It contains no TypeScript source, no runtime JavaScript that runs inside VS Code, and no language server. Its entire function at runtime is to supply file-type associations, TextMate grammars, and editor configuration to the VS Code extension host via the `contributes` keys in `package.json`.

**Grammar derivation chain.** All four shipped grammars originate from two upstream repositories. The base `JSON.tmLanguage.json` is fetched verbatim. The three derived grammars are produced by `adaptJSON` in `update-grammars.js`: it performs a recursive in-place rename of all `.name` and `.contentName` scope strings, rewriting the `.json` (or `.json.comments`) suffix to the target dialect's suffix (`.json.comments`, `.json.lines`, or `.json.comments.snippets`). This means the four grammars share identical structural token rules; they differ only in scope-name strings, which downstream theme and semantic token consumers use to apply styling.

**Shared language-configuration.** All four language IDs reference the single `language-configuration.json`. This means bracket matching, auto-close, indentation, and on-enter rules are uniform across `json`, `jsonc`, `jsonl`, and `snippets`. The `comments` block declares `//` and `/* */` even though plain JSON does not support comments; this is intentional because the `json` language ID in VS Code has historically tolerated JSONC-style comments in some contexts.

**Scope name hierarchy.** The scope chain `source.json` → `source.json.comments` → `source.json.lines` / `source.json.comments.snippets` is significant for TextMate theme resolution: a theme rule targeting `source.json` will apply to all four dialects due to scope prefix matching. Rules targeting `source.json.comments` apply to both JSONC and snippets but not plain JSON. The snippets grammar sits at the deepest level (`source.json.comments.snippets`), making it the most specifically overridable.

**No TypeScript/runtime code.** The extension contributes no `activationEvents`, no `main` entry point, and no commands. It is a grammar-only extension. All language intelligence (validation, completion, hover) for JSON in VS Code is provided by a separate extension, `vscode-json-languagefeatures` (not in this partition).

**Build vs. runtime boundary.** `update-grammars.js` and `cgmanifest.json` are excluded from the shipped VSIX (via `.vscodeignore`). The grammar `.json` files under `syntaxes/` are pre-built artifacts checked into the repository; they are what ships.

**Prose summary.** The `extensions/json` partition is a self-contained, runtime-passive VS Code extension that provides syntax highlighting and basic editor mechanics for four JSON dialects: plain JSON, JSON with Comments (JSONC), JSON Lines (JSONL), and VS Code Snippets files. It contains no TypeScript source and no activation logic. At build time, `update-grammars.js` fetches two upstream TextMate grammars via `vscode-grammar-updater`, producing four `.tmLanguage.json` files under `syntaxes/`; three of the four are scope-renamed derivations of the base JSON grammar, produced by the `adaptJSON` function which recursively rewrites all scope-name strings to carry the appropriate dialect suffix. At runtime, `package.json` declares the four language IDs with their file-extension and filename associations, pairs each language to its grammar file and scope name, and points all four to a shared `language-configuration.json` that defines bracket matching, auto-close pairs, indentation rules, and an on-enter rule for continuing `//` line comments. The shipped VSIX contains only the `syntaxes/` grammars, the `language-configuration.json`, the `package.json`, and `package.nls.json`; the build script and component manifest are excluded. All runtime language intelligence for JSON (schema validation, completion, hover) is handled by a separate extension outside this partition.

#### Pattern 5: Grammar Build-Time Adaptation
**Where:** `extensions/json/build/update-grammars.js:9-32`
**What:** Transforms base JSON grammar through scope name replacement to create variant grammars (JSONC, JSONL, Snippets) via recursive object traversal.
```javascript
function adaptJSON(grammar, name, replacementScope, replaceeScope = 'json') {
  grammar.name = name;
  grammar.scopeName = `source${replacementScope}`;
  const regex = new RegExp(`\\.${replaceeScope}`, 'g');
  var fixScopeNames = function (rule) {
    if (typeof rule.name === 'string') {
      rule.name = rule.name.replace(regex, replacementScope);
    }
    if (typeof rule.contentName === 'string') {
      rule.contentName = rule.contentName.replace(regex, replacementScope);
    }
    for (var property in rule) {
      var value = rule[property];
      if (typeof value === 'object') {
        fixScopeNames(value);
      }
    }
  };
  var repository = grammar.repository;
  for (var key in repository) {
    fixScopeNames(repository[key]);
  }
}
```
**Variations:** Deep tree walk using property iteration; scope replacement via regex with dynamic scope suffix (e.g., .json.comments, .json.lines).

## Porting Implications

The JSON extension demonstrates several critical VS Code IDE patterns that would require architectural changes for Tauri/Rust:

1. **Extension Manifest System**: The package.json contributes pattern requires a runtime contribution system that dynamically registers languages, file associations, and grammars. A Tauri-based IDE would need either plugin-like binary extension support or a declarative configuration loader.

2. **TextMate Grammar Integration**: VS Code's entire syntax highlighting pipeline depends on TextMate grammar (tmLanguage.json) compilation and scope-based theme application. Porting would require either embedding a TextMate grammar engine in Rust (e.g., using `tree-sitter` or similar) or maintaining a grammar interpretation layer.

3. **Dynamic File Association**: The multi-level file discovery (extensions, filenames, filenamePatterns via glob) would map to Rust pattern matching, but the runtime flexibility of glob patterns and dynamic registration would need a maintained glob library and refresh mechanism.

4. **Language Configuration Metadata**: The declarative rules for auto-closing, bracket matching, and indentation via regex patterns exist across extension boundaries. Porting requires either: (a) moving all rules into core Rust code with compiled regex, or (b) building a rule engine that interprets JSON configuration at runtime.

5. **Build-Time Grammar Transformation**: The update-grammars.js script shows that VS Code extensions expect to perform build-time transformations on grammars. A Tauri port would need to either provide equivalent build tooling or move grammar processing entirely to compile-time procedures.

6. **External Grammar Sourcing**: The vscode-grammar-updater dependency shows extensions assume the ability to pull external resources during installation/build. This would require network access policies, caching, and versioning mechanisms in a Rust-based system.
